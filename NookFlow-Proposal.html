NookFlow
让每场会议和学习任务都拥有自己的刘海工作台

NookFlow 是 X Nook 的任务上下文工作流：从日历事件开始，在刘海面板中关联笔记和文件架，结束后留下最近任务记录。它把 MacBook 刘海屏从被动展示区，变成不打断当前桌面的轻量工作入口。

01 / 创意简介

把刘海屏变成任务的入口，而不是通知的终点

MacBook 的刘海是一个始终靠近视线、却很少承载工作上下文的空间。NookFlow 让这块区域与当前的会议或学习任务同步——开始任务时它展开为工作卡片，结束时它把过程沉淀为可回看的记录。

基础项目

X Nook 已实现 macOS 刘海工具中心：日历、便签、文件托盘、媒体控制、双指手势切换、主题与设置。但这些能力目前彼此独立，缺少围绕"一次任务"的串联。

NookFlow：把能力串成一条任务链路

参赛版本不把日历、笔记和文件架堆成更多入口，而是把它们组织为同一条任务生命周期：选择日程 → 开始任务 → 关联资料 → 结束 → 查看最近任务。

02 / 目标用户与痛点

反复切换应用，是注意力的隐形杀手

核心用户为使用 MacBook 的学生与办公人群。他们在上课、开会或深度学习时，需要频繁在日历、笔记、Finder 与多个应用之间切换，临时资料难以与具体任务关联，结束后难以回溯。

切换成本高

日历看时间、笔记记内容、Finder 翻资料、聊天工具对日程——一次任务要打开 4-5 个应用，注意力被反复打断。

资料与任务脱节

会议中临时下载的文件、随手记的要点，散落在桌面和多个笔记里。一周后想找"上次那个会议的设计稿"，往往要翻半小时。

缺少任务边界感

没有明确的"开始 / 结束"仪式，任务之间的上下文互相串味。结束会议后没有沉淀环节，待办和关键决策容易被遗忘。

04 / 核心功能

已经交付的核心能力

NookFlow 以“任务上下文”而非“功能堆叠”为中心：日历、笔记与文件架各自保留原有用途，但在一次任务里形成可验证的闭环。

任务上下文卡片

从日历事件开始后，刘海展开为任务卡片：任务标题、开始时间、已进行时长、关联笔记入口、关联文件架入口与结束任务操作集中在同一处。

日程驱动入口

日历 Widget 不再只用于查看。用户可从日历事件一键开始任务，事件标题和结束时间成为会话上下文，让“看到日程”自然接上“开始处理”。

关联笔记

进行中的任务可一键创建并关联一条笔记。关联后卡片直接显示笔记标题并禁用重复创建，明确告诉用户这条任务已经有记录入口。

文件架关联

用户先将资料放入文件架，再将整个文件架关联到当前任务。系统对已关联文件去重；文件架为空时按钮会提示并禁用，避免无效操作。

可控的最近任务

结束任务后，它会按结束时间沉淀为最近任务。用户可在设置中选择显示 1–5 条，显示更多记录时展开面板会按行数增加高度，不压缩其他组件。

轻量回顾摘要

每条最近任务显示任务名称、关联笔记 / 文件摘要和结束时间。它不是又一个复杂的项目管理系统，而是帮用户快速找回“刚才在处理什么”。

05 / 价值与意义

让任务有边界，让上下文可回看

NookFlow 的价值不在于"又一个笔记应用"，而在于把分散的工具按任务串联起来，给学生和办公人群一个低成本的上下文沉淀方式。

一步

减少重新定位

从“先找日历、再找笔记、再找资料”的多次定位，收敛为从当前日程直接开始任务。它不承诺夸张的效率数字，只减少一段真实存在的启动摩擦。

1 个

资料与任务绑定

每次任务都有自己的笔记和文件关联摘要。回溯时，用户先找到任务，再回到资料，而不是从文件夹和窗口中反向猜测任务。

1–5

刚刚好的回顾密度

用户决定展示最近几条任务，面板随条目增长而伸展。对于小而高频的刘海空间，克制的信息密度本身就是体验的一部分。
