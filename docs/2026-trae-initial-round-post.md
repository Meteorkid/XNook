# 【学习工作】X Nook：把 MacBook 刘海变成任务上下文工作台

> **先体验，再阅读。**
>
> - 网页交互演示（无需安装）：<https://meteorkid.github.io/XNook/NookFlow-Proposal.html>
> - macOS 完整体验（X Nook v1.3.11）：<https://github.com/Meteorkid/XNook/releases/tag/v1.3.11>
> - DMG 直达下载：<https://github.com/Meteorkid/XNook/releases/download/v1.3.11/XNook-1.3.11.dmg>

## 一、我想解决的不是“少一个工具”，而是“少一次重新进入工作”的过程

对使用 MacBook 的学生和办公人群来说，开始一段任务前往往会发生一串微小却高频的切换：打开日历确认安排、找到笔记、翻出 Finder 里的资料、再回到正在做的事。

真正消耗注意力的，常常不是任务本身，而是反复回答“我现在要做什么、资料在哪、刚才做到哪”的重新定位。

**X Nook** 是一款面向 macOS 14+ 的刘海工具中心。NookFlow 是我在它之上完成的任务上下文工作流：不再把日历、笔记、文件架做成彼此独立的快捷入口，而是让它们围绕同一件正在做的事自然串起来。

它不试图替代大型任务管理、笔记或项目协作软件；它只负责一个更克制的目标：**当用户准备开始一件事时，在不离开当前桌面的前提下，把正确的上下文放到手边。**

## 二、核心创意：把“刘海”从硬件边界，变成任务的状态入口

MacBook 刘海处于用户的视线中心附近，却极少承载正在进行的工作。NookFlow 利用这块小而高频的区域，为一次学习、会议或办公任务提供一个明确的开始、进行和结束状态：

```text
日历事件
   ↓ 开始任务
NookFlow 任务卡片
   ↓ 关联笔记 / 文件架
结束任务
   ↓
最近任务（标题、关联摘要、结束时间）
```

这条链路的创新不在于叠加更多模块，而在于三个产品取舍：

1. **以任务而不是应用为组织单位**：用户不必先判断“该打开日历、笔记还是文件”，而是从当前日程直接进入同一张任务卡片。
2. **把资料关联做成低成本动作**：笔记创建后立即关联；文件先进入已有文件架，再一键写入当前任务快照。用户保留原有使用习惯，却获得任务层面的上下文。
3. **尊重刘海空间的有限性**：最近任务只显示用户选择的 1–5 条；每增加一条，展开面板按行数增高，而不是挤压日历、媒体等已有组件。

## 三、真实可体验的核心链路

### 1. 从日程开始，而不是从空白待办开始

日历 Widget 中的“开始任务”把一个日程事件转成 NookFlow 会话，并写入事件标题和计划结束时间。对于学生，它可以是一节课或一次复习；对于办公用户，它可以是一场会议或一段文档处理时间。

这一步把“看见时间安排”变成“开始处理这件事”，减少了任务启动时的断点。

### 2. 任务进行中，关联真正需要的资料

在任务卡片中：

- 点击“创建关联笔记”，系统创建并关联一条笔记；关联完成后显示笔记标题，并禁止重复创建。
- 将资料放入文件架后，点击“关联文件架”，系统将文件架中的文件快照关联到当前任务，并对已关联文件去重。
- 如果文件架为空，按钮会显示“文件架为空”且不可点击。这个细节很小，但它让界面在边界状态下仍然说清楚“下一步该做什么”。

这不是把文件和笔记“自动塞进”任务，而是让用户在需要时用一次明确动作把它们纳入任务上下文，避免误关联，也保留控制感。

### 3. 结束后，不做复杂管理，只留下足够的回忆线索

点击“结束任务”后，会话会按结束时间进入“最近任务”。每条记录展示任务名称、关联笔记 / 文件摘要和结束时间。用户可在设置中选择显示最近 1–5 条；面板高度随可见记录数增长，不遮挡其他组件。

这个设计刻意没有把 X Nook 变成另一个重型项目管理系统。它提供的是一个轻量答案：当用户想起“刚才那场会 / 那节课”，能快速找回它所关联的上下文。

【插入产品截图 3：结束任务后的“最近任务”列表；建议同时呈现设置中的 1–5 条显示选项，或多条记录下的自适应面板高度】

## 四、为什么它适合“学习工作”场景

**学生场景：**课前从日历开始“算法复习”或一节课程，过程里创建关联笔记、把讲义或作业资料放进文件架；结束后，最近任务保留这次学习的名称和关联线索。

**办公场景：**会议开始时从日程一键进入“产品周会”，把会议笔记和相关文档关联到会话；结束后，无需重新翻日历和 Finder，就能在最近任务中确认这次会议的上下文。

两个场景的共同点是：它们都不是需要复杂拆解的长期项目，而是需要**快速开始、少切换、可轻量回看**的一段具体工作。

## 五、完整性与体验边界

我希望评委能清楚地区分“已实现”与“仅设想”：

| 已可体验 | 本次不宣称 |
| --- | --- |
| 从日历事件开始任务 | AI 自动总结或生成任务 |
| 关联一条笔记、关联文件架中的资料 | 任务待办、跨应用自动采集 |
| 结束任务后展示最近任务摘要 | 历史筛选、统计分析、云端协作 |
| 最近任务 1–5 条设置与面板动态高度 | 用未经验证的效率提升百分比 |

我认为这种克制比“堆概念”更重要：每项写出的能力，都能在网页工作流或 macOS App 中找到对应验证路径。

## 六、TRAE IDE 如何参与这次创作

这次迭代不是一次性生成界面，而是沿着“需求拆解 → 数据建模 → SwiftUI 接入 → 边界修复 → 测试回归”的顺序推进。TRAE IDE 在下面三个阶段帮助我把产品想法落到可验证的实现上：

### 阶段一：定义任务上下文，而不是新增一个孤立 Widget

我先把需求收束为会话生命周期：任务标题、日程来源、开始 / 结束时间、关联笔记、关联文件快照，以及持久化后的历史记录。这样日历、笔记与文件架不需要互相直接依赖，只在任务边界上交换必要快照。
